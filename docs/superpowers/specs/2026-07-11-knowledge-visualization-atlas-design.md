## Purpose

Answer three questions for KennisBank:

1. How does a **human** currently access the wiki articles and look at the memory?
2. How could we **visualize** the knowledge, the memory, and the temporal
   dimension the system already models?
3. What do **comparable systems** do, and which of their ideas are worth taking
   given KennisBank's noord-ster (invisible, fast, local, KISS, retrieval-first,
   human-as-editor-in-chief)?

This document is the research synthesis and the design north-star. The
buildable work is broken down under EPIC TASK-27.

## 1. Current human access (the gap)

Every human-facing surface today is either a third-party GUI or plain text.
**The repository itself produces no visual artifact** â€” only markdown and
SQLite indexes.

| Access path | Tool | Human sees | Limitation |
|---|---|---|---|

| Obsidian graph view | native feature | node-link graph of wikilinks | Obsidian-only, not generated by the repo |
| Understand-Anything | external MIT plugin, `/understand-knowledge` | interactive knowledge-graph dashboard | requires installing a separate plugin; repo only emits `index.md`/`log.md` |

**Weakest spot: the memory layer.** `09-memory/` is richly structured

queue, supersede chains, or what has gone stale without grepping frontmatter.

## 2. What the data model already gives us

The data-model reference (confirmed from source) shows KennisBank already
carries every axis a strong visualization needs. The join key across all stores
is the **file path** (`docs.path` = `activity_events.source_path` = graphify
node `source_file` = usage `stem`).

- **Memory** (`09-memory/*.md`, `scripts/_memory.py`): `memory_type`
  (`feit`/`voorkeur`/`procedure`/`beslissing`), `importance` 1â€“5, `status`
  (`unverified`/`current`/`superseded`/`retracted`/`expired`), `evidence_basis`
  (trust tier), bi-temporal `valid_from`/`valid_until` vs `created`/`updated`,
  `superseded_by` links. Derived `provenance_tag` â†’ `mens-in-lus`/`autonoom`/
  `onbevestigd`.
- **Activity** (`.claude/kb-activity.db`, `scripts/_activity.py`):
  `activity_events` with `event_time` vs `captured_at`, `activity_kind`, derived
  lifecycle `state` (`released`/`blocked`/`fixed`/`superseded`/`changed`/
  `introduced`), `entities`, `topic_tags`, `artifacts`, `confidence`,
  `source_kind`, `unknown_time`. Entity/topic side tables for co-occurrence.
- **Index** (`.claude/kb-index.db`, `scripts/_kbindex.py`): `docs(path, layer,
  status, title)`, vector + FTS5, RRF fusion.
- **Ranking** (`scripts/_rank.py`): `relevance Ă— recency Ă— importance Ă— trust Ă—
  usage`, half-lives per memory type, one-hop graph neighbour.
- **Usage** (`.claude/kb-usage.db`, `scripts/_usage.py`): per-doc `injected`/
  `used`/`last_injected`/`last_used` â€” the warmth signal.
- **Graph** (`graphify-out/graph.json`, `02-wiki/index.md` + `log.md`):
  `nodes{id, source_file}`, `links{source, target, relation, confidence_score}`,
  optional `hyperedges`; Karpathy categories; chronological session log.

## 3. Comparable systems (research)

| System | Idea worth taking | Rejected / deferred |
|---|---|---|
| **Zep / Graphiti** | Bi-temporal graph with **time as a first-class axis**; ask "what was true as-of date X". Provenance to source episodes. | Mandatory graph DB / Neo4j. |
| **brain-map-skill** (vladignatyev) | A folder of markdown â†’ **one self-contained interactive HTML file**: force graph + **growth-timeline scrubber** + click-to-inspect. No server, no Obsidian. | Nothing â€” this is the target shape. |
| **agentmemory** viewer | **Memory browser + health dashboard + retrieval waterfall** (BM25 â†’ embed â†’ RRF â†’ rerank) that explains *why* something surfaced. | Live streaming server. |
| **YourMemory / Mem0** | Memory as **decay + importance + warmth**; eviction candidates. Maps to `_rank.py` + usage warmth. | Ebbinghaus auto-forgetting as mandatory deletion (KennisBank never silently deletes). |
| **Obsidian / Logseq / Scrintal** | Baseline node-link graph. | Being *only* a link graph â€” KennisBank's edge is time + belief-state. |

Throughline: the best systems add **time** and **belief-state** on top of a link
graph. That is exactly what KennisBank's data has and its current tooling
ignores.

## 4. Recommended concept: the KennisBank Atlas

**One self-contained HTML file**, regenerated off the hot path (a `/atlas`
command and an optional idle hook, in the shape of `daily_graphify`), written to
`<vault>/.claude/atlas.html`. No server, no CDN, no network, no Obsidian â€” it
opens over `file://`. One generator, one file, multiple lenses (KISS: one
mechanism, several views). This honours every noord-ster principle:

- **Local, always.** Nothing leaves the machine; a static file with inlined JS.
- **Performance first.** Generated at write/idle time; sub-second to open.
- **Retrieval-first & human-authority.** The lenses foreground the two
  mechanisms the README says matter: temporal validity and human-in-the-loop
  review.
- **KISS.** A single deterministic export + a single renderer.

### Lenses (priority order)

1. **Graph** â€” wiki + memory nodes; edges from wikilinks + graphify
   `relation`/`confidence_score` + per-claim provenance. Encoding: colour =
   `memory_type`/`layer`, size = `importance`, halo = usage warmth, border =
   `status`. Click-to-inspect panel with links back to source files.
2. **Time slider (the differentiator)** â€” scrub a date; the graph redraws to the
   **valid-as-of** state using `valid_from`/`valid_until`. Zep's temporal KG,
   rendered locally.
3. **Memory Health** â€” quarantine queue (`status: unverified`), supersede chains
   (`superseded_by` + `valid_until`), importance Ă— recency heatmap, stale/warm
   from usage. The editor-in-chief cockpit.
4. **Timeline** â€” bi-temporal horizontal timeline (event-time vs capture-time
   rows), coloured by `activity_kind`/`state`, with a range brush. The visual
   `/timeline`.
5. **Recall Inspector** â€” for one query, the retrieval waterfall: vector + FTS â†’
   RRF fusion â†’ rerank factors (`recency Ă— importance Ă— trust Ă— usage`) â†’
   graph-neighbour expansion. Turns `kb-eval`/`kb-calibrate` numbers into an
   explanation.
6. **Provenance / trust overlay** â€” overlay `kb-lint` coverage; flag unsourced
   claims (hallucination risk).

### Non-goals

- No hosted service, no mandatory graph database, no network binding.
- No dependency on Obsidian or the Understand-Anything plugin for the baseline.
- No "summarise everything" view without evidence links; every node/panel traces
  back to a source file/event.
